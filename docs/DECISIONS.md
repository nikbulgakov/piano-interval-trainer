# Журнал решений

Документ хранит причины решений, которые иначе пришлось бы восстанавливать из истории чата.

## D-001. Статическое приложение без backend

- Статус: принято.
- Решение: вся логика работает локально в браузере.
- Причина: MIDI доступен через браузерный API; аккаунты, синхронизация и сервер не нужны для MVP.
- Следствие: простой запуск, дешёвая публикация, MIDI-данные не покидают устройство.

## D-002. React + TypeScript + Vite

- Статус: принято.
- Решение: использовать распространённый минимальный SPA-стек без дополнительных framework-слоёв.
- Причина: несколько экранов и нетривиальное состояние сессии проще поддерживать в компонентах и reducer; TypeScript защищает музыкальные модели и MIDI-события.
- Отклонено: тяжёлый meta-framework, SSR, отдельный backend, глобальный state manager.

## D-003. Web MIDI за адаптером

- Статус: принято.
- Решение: браузерный MIDI-код не импортируется доменной логикой.
- Причина: музыкальные правила и state machine должны тестироваться без физического устройства.
- Ограничение платформы: API требует безопасный контекст, пользовательское разрешение и может ограничиваться политикой браузера. Источник: [MDN Web MIDI API](https://developer.mozilla.org/en-US/docs/Web/API/Web_MIDI_API).

## D-004. Основная поддержка Chrome и Edge desktop

- Статус: принято для MVP.
- Решение: гарантировать основной ручной сценарий в актуальных Chromium-браузерах; в остальных проверять API во время выполнения.
- Причина: не следует имитировать MIDI или обещать одинаковую поддержку там, где браузер не предоставляет нужный API.

## D-005. Восходящий гармонический интервал

- Статус: принято пользователем.
- Решение: пользователь одновременно удерживает нижнюю исходную ноту и верхнюю ноту на точном расстоянии в полутонах.
- Причина: это наиболее прямое прочтение задачи «построить интервал на пианино» и даёт однозначную проверку.
- Не входит: нисходящие и последовательные мелодические ответы.

## D-006. Три фиксированные октавы

- Статус: принято пользователем.
- Решение: использовать MIDI 48–83; принимать ответы только внутри этого диапазона.
- Причина: экран и проверка совпадают, а любая задача до октавы имеет хотя бы одно допустимое положение.
- Компромисс: на большой MIDI-клавиатуре нажатия вне диапазона не участвуют в упражнении.

## D-007. Точное множество из двух клавиш

- Статус: принято пользователем.
- Решение: правильной является ровно одна допустимая пара; лишняя клавиша делает попытку неправильной.
- Причина: правило понятно и исключает случайное засчитывание правильных нот внутри кластера.

## D-008. Sustain не влияет на проверку

- Статус: принято для MVP.
- Решение: Control Change игнорируется; учитываются физические Note On/Off.
- Причина: задача тренирует нахождение клавиш, а не звучащие после педали ноты.

## D-009. Перемешанный мешок заданий

- Статус: принято.
- Решение: пройти все выбранные пары «нота × интервал» в случайном порядке до повторного перемешивания.
- Причина: равномерное покрытие полезнее независимого random, который может часто повторять одни задания.

## D-010. Документация для разных coding-агентов без дублирования

- Статус: принято.
- Решение: `AGENTS.md` — единый короткий набор постоянных правил; `CLAUDE.md` импортирует его; подробности вынесены в предметные документы.
- Причина: OpenAI рекомендует короткий практичный `AGENTS.md` и отдельные task-specific документы, Anthropic — конкретный и краткий `CLAUDE.md`; Claude Code официально поддерживает импорт `@AGENTS.md`.
- Источники: [OpenAI best practices](https://developers.openai.com/codex/learn/best-practices), [OpenAI AGENTS.md](https://developers.openai.com/codex/guides/agents-md), [Anthropic memory](https://code.claude.com/docs/en/memory).

## D-011. Один спринт — одна пользовательская приёмка

- Статус: принято.
- Решение: следующий спринт начинается только после ручной проверки пользователем.
- Причина: MIDI и ощущение интерфейса требуют реального устройства; небольшие этапы дешевле корректировать.

## D-012. Раздельные экраны настройки и практики

- Статус: принято пользователем.
- Решение: приложение имеет поток `главный экран → настройка выбранной тренировки → практика → результаты`.
- Причина: во время упражнения пользователь должен видеть только текущее задание, таймер, обратную связь, завершение и клавиатуру.
- Следствие: MIDI-подключение и параметры скрываются после старта, но их состояние сохраняется при возврате к настройкам.

## D-013. Абсолютное расписание timed-режима

- Статус: заменено решением D-014.
- Решение: дедлайн каждого следующего задания вычисляется прибавлением периода к предыдущему дедлайну, а не к моменту срабатывания UI-таймера.
- Причина: задания должны меняться строго по расписанию, даже после раннего ответа или задержки фоновой вкладки.
- Следствие: при возврате из фона приложение обрабатывает все прошедшие дедлайны; нерешённые задания учитываются как пропуски, а удержанные клавиши не засчитывают новое задание до полного отпускания.

## D-014. Немедленная смена timed-задания после правильного ответа

- Статус: принято пользователем.
- Решение: правильный ответ сразу показывает следующее задание и запускает для него новый полный период.
- Причина: пользователь не должен бездействовать после раннего правильного ответа.
- Следствие: нерешённые задания по-прежнему меняются по абсолютным дедлайнам и учитываются как пропуски; после правильного ответа расписание начинается заново от момента ответа.

## D-015. Отключение MIDI не приостанавливает сессию

- Статус: принято для локального MVP.
- Решение: при потере MIDI проверка ответов блокируется и показывается возврат к подключению, но общий таймер продолжает идти от исходного абсолютного дедлайна.
- Причина: автоматическая пауза усложнила бы scoring и timed-расписание; явный возврат позволяет быстро восстановить устройство или начать новую сессию.

## D-016. Cloudflare Pages с GitHub-интеграцией

- Статус: отменено решением D-021.
- Историческое решение: публиковать production из `main` в Cloudflare Pages.
- Причина на момент принятия: статическое Vite SPA не требовало сервера, а GitHub-интеграция автоматически обновляла резервный production после merge.
- Использовавшаяся конфигурация: `npm run build`, каталог `dist`, Node.js 24.
- Доступ GitHub App Cloudflare был ограничен репозиторием `piano-interval-trainer`.

## D-017. Vercel как основной публичный хостинг

- Статус: принято пользователем.
- Решение: публиковать `main` в Vercel и использовать `https://piano-interval-trainer.vercel.app` как единственный публичный адрес.
- Причина: Vercel доступен пользователю без VPN и закрывает production- и preview-сценарии проекта.
- Конфигурация: Vite, `npm run build`, каталог `dist`, Node.js 24.x.
- Следствие: каждый pull request должен пройти lint в GitHub Actions и получить Vercel Preview; merge в `main` обновляет production.

## D-018. Глобальные настройки обозначений

- Статус: принято пользователем.
- Решение: одна настройка выбирает русские или латинские обозначения для всех нот, вторая — полные названия или краткие символы для всех интервалов.
- Значения по умолчанию: русские названия нот и полные названия интервалов.
- Следствие: смешанные подписи вроде `До (C)` и `малая терция (m3)` не используются; выбор применяется во всех экранах и сохраняется в `localStorage`.
- Граница: музыкальная логика по-прежнему опирается на pitch class и число полутонов, а не на отображаемые строки.

## D-019. Проверка feature-веток через lint и Vercel Preview

- Статус: принято пользователем.
- Решение: каждый спринт ведётся в отдельной ветке и draft PR; обязательная автоматическая проверка — только lint, а ручная приёмка выполняется на Vercel Preview.
- Решение по хостингу: feature-ветки получают только Vercel Preview.
- Следствие: merge выполняется только после явного подтверждения пользователя и ручной проверки с MIDI-клавиатурой для затронутых сценариев.

## D-020. Главный экран как каталог тренировок

- Статус: принято пользователем.
- Решение: приложение сначала показывает отдельный экран выбора типа тренировки; настройка MIDI и параметров открывается только после выбора режима.
- Текущий этап: интервалы доступны, а поиск одиночных нот показан недоступной карточкой «Скоро» до реализации полноценного сценария.
- Причина: структура сразу поддерживает несколько упражнений, но пользователь не попадает в пустой или частично работающий режим.
- Техническое следствие: навигация остаётся локальной машиной состояний в `App`, без router и глобального state manager; MIDI-состояние и конфигурация не сбрасываются при возврате домой.

## D-021. Полный отказ от Cloudflare Pages

- Статус: принято пользователем.
- Решение: удалить Pages-проект `piano-interval-trainer`, его deployment history и GitHub-интеграцию Cloudflare; Vercel оставить единственным хостингом.
- Причина: Cloudflare недоступен пользователю без VPN и создаёт лишний второй production-процесс без практической пользы.
- Следствие: адрес `piano-interval-trainer.pages.dev` больше не поддерживается; документация и выпуск изменений описывают только Vercel.
